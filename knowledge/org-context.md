## The Membership Program (MOST IMPORTANT CUSTOMIZATION)

**POW does NOT use NPSP membership fields/rollups.** "Team POW" membership is
fully custom: a constituent is a current member if they did ANY of these in the
last 365 days — donation (any Opp record type), POW opt-in (Formstack), partner
opt-in (Ikon/Aspen/etc. via Formstack), merchandise purchase, event attendance
(Blackthorn), or advocacy action (Quorum/P2A campaign).

- `Membership_Status` is a **formula field** on `Last_Membership_Date_v2__c`
  (Current if within 365 days, else Expired).
- Membership fields (entry points, join date, last/end dates, amounts) are
  populated by **five custom Flows** ("Membership Flow - …" family — see
  ORG_AUTOMATION.md for exact API names).
- **Do not** write to membership fields directly from new automation, and do
  not introduce NPSP membership objects. Any task touching membership behavior
  must account for all five flows.

## Donor Levels

Assigned via the Salesforce standard **Levels** object (NPSP Levels), based on
18-month giving: Ally <$1k, Advocate $1k-5k, Enthusiast $5k-10k, Champion
$10k-25k, Titan >$25k. `Previous Donor Level` tracks the prior level.
Separate from membership. Contact Stages (Current/Lapsed) also use an
18-month window via the `Contact Stages - v2` flow.

## Record Types That Change Behavior

- **Contact:** `Contact` (general) vs `Alliance Member` (athletes/scientists/
  creatives). Alliance members are excluded from Cloudingo merges, the data
  retention archive, and several automations. Always ask which record types an
  automation should target.
- **Account:** Household (NPSP auto-created), Organization, Partner. Partner
  accounts carry contract fields (Partner Level, Contract Start/Amount,
  Renewal Date) and stage workflows (Current/Lapsed/Lost on 365/730-day
  donation recency).
- **Opportunity:** Donation, Classy, Event Ticket, Grant, In-Kind, Matching
  Gift, Merchandise, Partner Commitment. "Opportunity" ≠ just donations here —
  merchandise and event tickets flow through it too.

## Data Operations OUTSIDE Salesforce (Cloudingo)

Cloudingo, not Flow/Apex, performs scheduled destructive data ops:
- Weekly contact merges (email+name match; excludes KnowWho + Alliance)
- Campaign deletion: no-member campaigns >90 days old (except Event, Classy,
  P2A campaigns)
- Mailchimp activity >30 days deleted; Formstack submissions >3 years deleted
  (except Alliance forms)

A weekly scheduled Flow sets `Archived__c = TRUE` per the retention policy
(no gifts, no affiliations, opted out, no membership/advocacy/events, >3 years
old). **Implication for builds:** never key automation off data Cloudingo may
delete, and never assume campaign or activity history is complete.

## Payments & Reconciliation (Resin-owned process)

- Classy → Classy Pay / PayPal; Blackthorn + WooCommerce → Stripe.
- Payment date = payout date convention for Classy/Stripe reconciliation.
- **Stripe payouts are reconciled monthly in Salesforce by Resin.** Tasks
  mentioning reconciliation are operational, not build, work.

## Voter / Advocacy Data (L2)

Contact carries ~25 voter fields (election voting methods 2014-2020, party,
districts, ballot tracking, propensity scores) loaded from L2 voter data
(2022 election), plus an `L2 Data` custom object. This data is static/imported
— do not build automation that writes to it.

## Naming Conventions (observed)

- Flows: descriptive names with area prefix ("Membership Flow - …",
  "Engagement - …", "Account - …"); follow the existing family pattern
- Custom fields: descriptive, no client prefix (e.g. `Last_Membership_Date_v2__c`)
  — note the `_v2` pattern when a field was rebuilt; check for v1 remnants
- Versioned flow names ("… 2", "… 3") indicate rebuilds — always target the
  active version listed in ORG_AUTOMATION.md

## Things That Break / Known Constraints

- Membership flows are interdependent — a change to one entry-point flow can
  silently skew Membership Status org-wide.
- DLRS rollups (14) + NPSP rollups + workflow field updates can stack on the
  same records; bulk metadata deploys touching Contact/Account fields should
  list which rollups read/write them in the PR.
- Duplicate rules: 4 contact + 1 account rule active in Salesforce, plus
  Cloudingo filters that mirror them — changing matching fields requires
  updating both layers (Cloudingo is manual/admin-side, flag in PR).
- `woocommerce` integration is marked "????!" in the client manual — treat
  merchandise sync details as unverified; confirm with Joe before building
  against Order ID or merchandise fields.
- The client manual's sections on report inventory, profiles/roles, email
  templates, and engagement plans were unfinished — ORG_REPORTS.md and
  ORG_SECURITY.md (auto-generated) are the only reliable sources for those.
